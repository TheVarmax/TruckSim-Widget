# Структура хранения данных заказов

## Описание системы

Программа теперь использует **систему отдельных файлов для каждого заказа**, что позволяет:

1. **Организованное хранение** - каждый заказ имеет свой файл
2. **Разделение по играм** - заказы ETS и ATS хранятся в отдельных папках
3. **Легкое управление** - удаление, чтение отдельных заказов
4. **Масштабируемость** - легко добавлять новые заказы

## Структура папок

```
C:\Users\<username>\AppData\Local\TruckSimWidget\
│
├── ETS/                          # Папка для данных Euro Truck Simulator 2
│   ├── state.json               # Основное состояние приложения (текущий заказ, позиция окна и т.д.)
│   ├── job_1778553643.json      # Заказ с уникальным ID 1778553643 из логов TB
│   ├── job_1778553644.json      # Заказ с уникальным ID 1778553644
│   ├── job_1778553645.json
│   └── ...
│
├── ATS/                          # Папка для данных American Truck Simulator
│   ├── state.json               # Основное состояние приложения для ATS
│   ├── job_1778553428.json      # Заказ с уникальным ID 1778553428
│   ├── job_1778553429.json      # Заказ с уникальным ID 1778553429
│   └── ...
│
└── state.dat                     # Основной файл состояния окна приложения
```

## Формат файла заказа

**Имя файла:** `job_<jobId>.json`

**Пример:** `job_1778553643.json`

**Содержимое файла:**

```json
{
  "TelemetryId": "1778553643",
  "DrivenDistance": 245.5,
  "MaxSpeedKmh": 120,
  "IsRace": false,
  "CargoWasLoaded": true
}
```

**Поля:**

- **TelemetryId** (string) - Уникальный ID заказа из логов TrucksBook
- **DrivenDistance** (float) - Пройденное расстояние с грузом (в км или милях)
- **MaxSpeedKmh** (int) - Максимальная скорость, достигнутая при доставке этого заказа
- **IsRace** (bool) - Был ли заказ выполнен в режиме "гонка" (скорость > 100км/ч или 81mph)
- **CargoWasLoaded** (bool) - Был ли груз физически загружен (true = активный заказ)

## Пример заполненного файла

```json
{
  "TelemetryId": "1778553643",
  "DrivenDistance": 587.3,
  "MaxSpeedKmh": 125,
  "IsRace": true,
  "CargoWasLoaded": true
}
```

## Процесс работы

### 1. Начало нового заказа

1. Игра отправляет телеметрию о новом заказе
2. TrucksBook логирует: `- Delivery exist, continue 1778553643 (roma > roma)`
3. Программа парсит ID `1778553643` из лога
4. Проверяет файл `job_1778553643.json` в папке ETS или ATS (в зависимости от текущей игры)
5. Если файл существует - загружает сохраненные данные
6. Если файла нет - инициализирует новый заказ с нулевыми значениями

### 2. Прохождение заказа

1. При загрузке груза: `CargoWasLoaded` становится `true`
2. Во время доставки отслеживаются:
   - `DrivenDistance` - растет при движении с грузом
   - `MaxSpeedKmh` - обновляется при превышении текущего максимума
   - `IsRace` - устанавливается в `true` если скорость > 100км/ч (ETS) или 81mph (ATS)
3. Данные сохраняются в файл при каждом обновлении

### 3. Завершение заказа

1. TrucksBook логирует: `- Delivered` или `- Cancelled`
2. Программа удаляет файл `job_1778553643.json` из папки игры
3. Заказ удаляется из памяти приложения

## Защита от путанницы между игрой

**Важно:** Программа определяет текущую игру из логов TrucksBook:

```csharp
if (logGame == GameType.Ats)
{
    // Используется папка ATS/
    latestDeliveryIdAts = extracted;
}
else
{
    // Используется папка ETS/
    latestDeliveryIdEts = extracted;
}
```

Это обеспечивает:
- ✅ Одинаковые ID заказов из разных игр не смешиваются
- ✅ Каждый заказ сохраняется в правильной папке
- ✅ При переключении между играми используются правильные данные

## Преимущества системы

1. **Масштабируемость** - не нужно загружать все заказы в памяти
2. **Надежность** - каждый заказ сохраняется отдельно
3. **Простота анализа** - легко посмотреть логи отдельного заказа
4. **Быстрота** - нет необходимости парсить большие файлы
5. **Гибкость** - легко добавить статистику или экспорт

## Миграция со старой системы

При первом запуске программы с новой версией:

1. Старые данные из `state.json` читаются нормально
2. При сохранении каждый активный заказ автоматически создается в отдельном файле
3. Старые файлы `state.json` остаются для обратной совместимости
